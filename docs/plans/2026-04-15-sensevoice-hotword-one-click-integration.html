SenseVoice Hotword One-Click Integration Implementation Plan ​

For Claude: REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

Goal: 让 VILab 用户可以在设置页里一键下载、激活并使用 dengcunqin/SenseVoiceSmall_hotword，同时不破坏现有 Whisper 和内置 SenseVoice 流程。

Architecture: 不建议让桌面客户端直接依赖 ModelScope 的原始快照结构或 snapshot_download。推荐改成“VILab 受控模型包 + 受控运行器”的两段式架构：客户端只认一个稳定的版本化安装包，下载后校验、解压、激活；真正的热词推理由单独的 sensevoice-hotword 运行边界负责，首版采用 sidecar runner 以最小风险接入作者现有 ONNX 目录结构，后续再视情况原生化到 Rust/ORT。

Tech Stack: Tauri 2, Rust, reqwest, tar/flate2, 现有 local_stt 管理器, React/TypeScript 设置页, 版本化模型包脚本, sidecar runner, SHA-256 校验。

推荐方案 ​

推荐路线：受控模型包 + sidecar runner ​

这是最合理的落地路径，原因如下：

  1. 这个模型不是当前 VILab 已支持的单文件 .bin，而是由多个 ONNX 子模块和配套 Python 封装组成。
  2. VILab 当前本地 STT 运行时只支持：
    • whisper.cpp 单文件模型
    • 现有 transcribe-rs 支持的 SenseVoice 目录模型
  3. 该 ModelScope 模型页明确展示了多文件依赖和自定义调用方式，不适合把“网页上的文件结构”直接硬编码进桌面客户端。
  4. 直接让客户端调用 ModelScope 的 snapshot_download 会引入：
    • 平台协议变动风险
    • 下载鉴权 / 镜像稳定性问题
    • 客户端安装结果不可控
    • 无法稳定做校验、回滚和客服排障

推荐把用户面向的一键下载抽象成：

  • 一个 VILab 自己维护的版本化模型包
  • 一个与模型包版本绑定的运行器包
  • 客户端只做“下载、校验、安装、激活、调用、回退”

不推荐路线 ​

  1. 客户端直接对接 ModelScope 快照

    • 好处：最少前期脚本工作
    • 问题：稳定性最差，无法保证长期一键安装体验

  2. 一开始就全量原生重写到 Rust/ORT

    • 好处：最终形态最干净
    • 问题：前期研发成本高，调试链路长，容易把现有本地 STT 稳定性一起拖慢

建议的阶段划分 ​

  1. Phase 1：先可用

    • 用 sidecar runner 跑通一键下载、激活和转写
    • 热词输入先复用 VILab 现有词库

  2. Phase 2：再原生化

    • 若效果和留存都成立，再评估改为 Rust + ONNX Runtime 原生运行

目标范围 ​

本次要解决 ​

  • 在设置页新增一个可下载的内置模型项，例如 SenseVoice Hotword
  • 一键下载时自动安装所需模型包和运行器
  • 激活后桌面录音转写和内嵌 HTTP /v1/audio/transcriptions 都能走这个模型
  • 复用 VILab 现有词库 / 热词表，把启用项组装成热词输入
  • runner 不可用时给出明确错误，并能回退到已有本地模型或云端流程

本次不解决 ​

  • 首版不要求直接从 ModelScope 官方页面实时拉取快照
  • 首版不要求改写作者的 ONNX 推理逻辑为纯 Rust
  • 首版不做“热词权重编辑器”
  • 首版不做多 runner 并存调度

安装包契约 ​

客户端不应直接依赖 ModelScope 页面结构。建议定义一个受控安装包契约：

模型包 ​

建议产物名：

text
sensevoice-hotword-small-v1.tar.gz

解压后目录建议固定为：

text
models/
  sensevoice-hotword-small/
    manifest.json
    model.onnx
    sensevoice_model_hot_emb.onnx
    sensevoice_model_hot_module.onnx
    sensevoice_model_nohot_module.onnx
    tokens.json
    README.origin.md
    LICENSE

manifest.json 至少包含：

json
{
  "model_id": "sensevoice-hotword-small",
  "engine": "sensevoice-hotword",
  "package_version": "v1",
  "source": "modelscope:dengcunqin/SenseVoiceSmall_hotword",
  "sha256": {
    "model.onnx": "..."
  },
  "runner_version": "v1",
  "supports_hotwords": true
}

运行器包 ​

建议产物按平台发布：

text
sensevoice-hotword-runner-darwin-arm64-v1.tar.gz
sensevoice-hotword-runner-windows-x64-v1.zip

客户端安装位置建议固定为：

text
<models_dir>/runners/sensevoice-hotword/<platform>/<version>/

数据流设计 ​

下载与安装 ​

  1. 设置页点击“下载模型”
  2. Rust 侧识别该 builtin model 需要：
    • 模型包
    • runner 包
  3. 依次下载并校验 SHA-256
  4. 写入临时目录
  5. 安装完成后原子替换目标目录
  6. 刷新模型列表并更新 UI 状态

激活与运行 ​

  1. 用户激活 sensevoice-hotword-small
  2. 本地运行时读取模型 manifest
  3. 检查 runner 是否存在且可执行
  4. 从词库中提取启用的热词 surface
  5. 组装为 runner 请求
  6. runner 返回 transcript 文本
  7. 若失败：
    • 记录错误
    • 状态设为 error
    • 给出“重新激活 / 切回其他模型”的 UI 提示

热词来源 ​

首版不新增独立配置面板，直接复用现有字典 / 热词表：

  • 读取 sqlite::load_dictionary(...)
  • 只选 enabled=true 项
  • 按语言过滤
  • 使用 phrase 或标准 surface 字段组装热词字符串
  • 对总长度和数量做上限保护，例如：
    • 最多 100 条
    • 总字符数不超过 1024

验收标准 ​

  • 设置页能看到 SenseVoice Hotword 模型项
  • 用户点击下载后，模型包和 runner 包都能自动安装
  • 激活后桌面听写可用
  • /v1/audio/transcriptions 在 local 模式下也可用
  • 启用词库时，runner 能收到热词字符串
  • runner 或安装异常时，错误信息可见且不会破坏已有模型
  • 现有 Whisper / SenseVoice 功能无回归

风险与缓解 ​

  1. ModelScope 原始目录变动

    • 缓解：客户端不直接依赖原始目录；只依赖 VILab 打包产物

  2. sidecar 体积过大

    • 缓解：runner 单独版本化和缓存，不跟主 app 更新强绑定

  3. 热词字符串过长导致推理变慢

    • 缓解：限制数量和总长度，必要时按频次截断

  4. 跨平台 runner 行为不一致

    • 缓解：runner 协议 JSON 化，E2E 用同一组样本音频做 smoke test

  5. 未来原生化重构成本

    • 缓解：从第一天起把 runner 封装在独立模块里，不让 UI 和业务层直接感知实现细节

结论 ​

这项改造的关键不是“把一个新的下载链接塞进 builtin 列表”，而是把 VILab 的本地模型体系从“单一文件/目录安装”提升为“可安装模型包 + 可安装运行器 + 可声明能力”的通用框架。只有这样，这个 ModelScope 热词模型才能真正做到稳定的一键下载和长期可维护使用。
