VILab 语音相似热词纠错方案 ​

日期：2026-04-13 状态：已定版，可进入实现

背景 ​

当前 VILab 的词典能力更偏向文本替换，主要在转录后的 smart clean 和 polished 阶段生效。对于中文专有名词、术语、人名等场景，单纯依赖文本后处理仍然不够稳，尤其是当 ASR 已经把词识别成另一个合法词语时，后续纯文本规则很难稳定纠回。

本方案将新增一条位于 ASR raw transcript 之后、smart clean / polished 之前的热词纠错链路。其目标不是做通用中文纠错，而是做一条“热词驱动、只看语音相似度、由 LLM 二选一裁决”的定向纠错链路。

已确认的核心决策 ​

  1. 候选召回阶段只考虑语音相似度，不考虑文字相似度。
  2. 本地阶段只为每个疑似片段找出一个 top1 热词候选，不做三选一。
  3. LLM 只在 原始 ASR 输出 和 热词标准词 之间做二选一。
  4. LLM 不负责自由改写，不负责生成新词，不修改候选片段以外的文本。
  5. 热词表不保留 aliases 字段。
  6. left context 和 right context 不单独作为 LLM 输入字段；整句上下文通过带标记的原句直接提供给模型。
  7. 系统内部仍保留 span start/end 偏移，用于替换、日志和调试。

目标 ​

本方案优先解决以下问题：

  • 中文专有名词识别错误
  • 术语识别错误，例如“大原模型”与“大语言模型”
  • 人名识别错误
  • 产品名、公司名、模型名识别错误

本方案不解决以下问题：

  • 通用中文语音纠错
  • 大规模开放域语义改写
  • 平翘舌、前后鼻音等通用音系混淆的系统性建模
  • 非热词区域的自动润色

整体流程 ​

运行时流程如下：

  1. ASR 返回 raw_transcript
  2. 进入 phonetic_hotword_resolver
  3. 对 raw_transcript 做局部片段扫描
  4. 将局部片段转为拼音表示
  5. 与热词表中的拼音表示做语音相似度匹配
  6. 对每个通过阈值的片段只保留一个 top1 热词候选
  7. 用带标记的整句将“原始片段 / 热词候选”送给 LLM
  8. LLM 对每一项只输出 KEEP 或 REPLACE
  9. 应用替换，得到 phonetic_resolved_text
  10. 将 phonetic_resolved_text 继续送入现有 verbatim / smart clean / polished 链路

重要约束：

  • raw_transcript 必须保留，不被覆盖
  • phonetic_resolved_text 作为后续文本处理的输入
  • 所有替换必须可追踪，可记录候选和最终决策

数据模型 ​

建议新增独立的热词表，而不是复用现有 dictionary_entries。

热词主表 ​

建议新增 hotword_entries：

sql
CREATE TABLE hotword_entries (
  id TEXT PRIMARY KEY,
  surface TEXT NOT NULL,
  category TEXT NOT NULL,
  enabled INTEGER NOT NULL DEFAULT 1,
  priority INTEGER NOT NULL DEFAULT 0,
  notes TEXT NOT NULL DEFAULT '',
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

字段含义：

  • surface：热词标准写法，例如 大语言模型
  • category：person / term / product / org
  • priority：小幅度排序加权
  • notes：辅助信息，例如“AI 术语”

热词读音表 ​

建议新增 hotword_pronunciations：

sql
CREATE TABLE hotword_pronunciations (
  id TEXT PRIMARY KEY,
  hotword_id TEXT NOT NULL,
  syllables_json TEXT NOT NULL,
  compact TEXT NOT NULL,
  is_primary INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (hotword_id) REFERENCES hotword_entries(id)
);
CREATE INDEX idx_hotword_pronunciations_compact ON hotword_pronunciations(compact);
CREATE INDEX idx_hotword_pronunciations_hotword_id ON hotword_pronunciations(hotword_id);

一个热词可对应多个读音。推荐存储格式如下：

json
{
  "surface": "大语言模型",
  "pronunciations": [
    {
      "syllables": ["da", "yu", "yan", "mo", "xing"],
      "compact": "dayuyanmoxing"
    }
  ]
}

设计说明：

  • syllables 用于保留音节边界，便于解释、调试和后续演进
  • compact 用于在线匹配，能更好兼容 yuan 与 yu yan 这类边界差异
  • 不仅要存去空格形式，也要保留音节数组；去空格形式不是唯一数据源，而是高频检索字段

拼音标准化 ​

本方案中的拼音标准化遵循以下规则：

  1. 全部转小写
  2. 默认去音调
  3. 保留音节数组
  4. 同时生成去空格的 compact 版本
  5. 同一热词允许人工维护多个读音

示例：

  • 大语言模型

    • syllables = ["da", "yu", "yan", "mo", "xing"]
    • compact = "dayuyanmoxing"

  • 语言

    • syllables = ["yu", "yan"]
    • compact = "yuyan"

去空格版本是本方案中最重要的在线匹配表示，因为它对连读、切分差异更鲁棒。

局部片段扫描 ​

本方案不拿整句直接与热词表匹配，而是对句子中的局部 span 扫描。

中文片段 ​

对中文文本采用滑窗扫描：

  • 最小长度：2 个汉字
  • 最大长度：8 个汉字

例如：

text
今天和张原讨论大原模型，之后发给李原看一下

会枚举局部片段，例如：

  • 张原
  • 讨论
  • 大原模型
  • 李原

ASCII / 混合片段 ​

对英文缩写、产品名、混合输入，补一层 token 片段扫描：

  • 最小长度：1 token
  • 最大长度：4 token

重叠处理 ​

本地召回后，如果多个候选片段重叠：

  • 先按分数降序保留
  • 再用“高分优先、长片段优先”的规则消除冲突
  • 冲突解决后再送 LLM

LLM 不应同时收到两个重叠候选。

语音相似度计算 ​

本方案只考虑语音相似度，不引入文字相似度。

输入表示 ​

对每个局部 span：

  1. 转成 syllables
  2. 转成 compact

对每个热词读音也已有对应的 syllables 和 compact

相似度公式 ​

建议初版只使用基于编辑距离的两项：

text
syllable_sim = 1 - levenshtein(join(span.syllables, " "), join(term.syllables, " ")) / max_len
compact_sim  = 1 - levenshtein(span.compact, term.compact) / max_len
score        = 0.35 * syllable_sim + 0.65 * compact_sim + priority_bonus

说明：

  • compact_sim 权重更高，因为其对 yuan 与 yu yan 等边界差异更鲁棒
  • priority_bonus 仅做很小加权，建议范围 0.00 ~ 0.05
  • 初版不考虑 zh/z、ch/c 等混淆规则

Top1 选择 ​

对每个局部 span：

  • 仅保留得分最高的一个热词候选
  • 若 score < recall_threshold，则放弃该片段，不送 LLM

建议初始阈值：

  • recall_threshold = 0.72

后续可用人工纠错数据调参。

LLM 裁决方案 ​

输入形式 ​

为了降低 prompt 复杂度，采用“整句内嵌标记”的方式，而不是复杂 JSON 输入。

示例：

text
今天和【1|张原|张源】讨论【2|大原模型|大语言模型】，之后发给李原看一下

其中：

  • 1：候选片段编号
  • 张原：原始 ASR 输出
  • 张源：热词表中召回的 top1

如果一句中有多个热词候选，可以在同一句中同时标注多个片段。

系统提示词约束 ​

LLM 的职责非常窄：

  1. 只对每个标记片段做二选一
  2. 只能选择 KEEP 或 REPLACE
  3. REPLACE 时只能替换成标记中的候选热词
  4. 不得生成新词
  5. 不得修改标记片段以外的任何文本
  6. 不确定时必须选择 KEEP

输出格式 ​

推荐输出稳定的结构化 JSON：

json
{
  "decisions": [
    {"id": "1", "action": "KEEP"},
    {"id": "2", "action": "REPLACE"}
  ]
}

虽然输入形式被简化为内嵌标记，但输出仍建议保持结构化，便于解析和审计。

多热词联合裁决 ​

本方案支持一句中出现多个热词候选。

原因是：

  • LLM 能看到整句语义
  • 各个标记项都位于同一上下文中
  • 适合处理“句子里同时包含人名和术语”的情况

因此：

  • 不需要单独传 left_context / right_context
  • 整句本身即为上下文
  • 每个片段通过唯一 id 进行绑定

VILab 内部集成点 ​

建议新增模块：

  • src-tauri/src/domain/hotword.rs

    • 拼音标准化
    • 局部片段扫描
    • 语音相似度计算
    • 标记句构造

  • src-tauri/src/infra/sqlite.rs

    • 新增热词表和读音表的 CRUD

  • src-tauri/src/application/mod.rs

    • 在 raw_transcript 生成后、smart_clean_output 之前插入 phonetic_hotword_resolver

推荐集成顺序 ​

现有主链：

text
raw_transcript
  -> verbatim
  -> smart_clean
  -> polished

调整后：

text
raw_transcript
  -> phonetic_hotword_resolver
  -> phonetic_resolved_text
  -> verbatim / smart_clean / polished

重要要求：

  • raw_transcript 仍然保存在会话里，作为真实原始输出
  • phonetic_resolved_text 仅作为后续文本链路的输入
  • 如需追踪，可在 debug artifacts 中记录每次候选与决策

调试与可观测性 ​

建议新增调试记录结构：

json
{
  "span_text": "大原模型",
  "start": 8,
  "end": 12,
  "candidate_surface": "大语言模型",
  "score": 0.91,
  "llm_action": "REPLACE"
}

建议记录：

  • 每次命中的局部片段
  • 召回的 top1
  • 分数
  • 是否送入 LLM
  • LLM 最终决策

这些数据将用于后续调阈值和评估误改率。

与人工纠错反馈的结合 ​

VILab 已经具备人工纠错反馈存储能力，可直接作为后续评估和热词维护基础。

当前已有：

  • CorrectionRecord
  • save_correction_feedback

这部分数据可用于：

  1. 评估热词纠错链路是否命中真实问题
  2. 发现值得加入热词表的新标准词
  3. 为热词补充新的读音变体

初版不做自动学习，仅保留“人工维护 + 离线分析”。

实施阶段 ​

V1 ​

目标：打通最小可用链路

  • 新增热词主表和读音表
  • 新增热词管理接口
  • 新增中文局部片段扫描
  • 新增基于拼音的 top1 召回
  • 新增 LLM 二选一裁决
  • 将结果接入 smart clean / polished 前
  • 增加基础调试日志

V2 ​

目标：提高稳定性和可维护性

  • 支持混合中英文片段
  • 优化重叠冲突处理
  • 增加热词优先级管理
  • 支持一词多读音的更完整编辑体验
  • 增加命中统计与评估工具

V3 ​

目标：建立持续优化闭环

  • 利用 corrections 数据做离线评估
  • 提供热词建议工具
  • 基于历史接受率自动调参

风险与边界 ​

主要风险 ​

  • 同音词过多导致错误召回
  • 人名读音不规则导致拼音化不稳定
  • 句中多个候选同时出现时，LLM 仍可能误判
  • 热词表维护不足时，命中率有限

当前控制策略 ​

  • 本地只做 top1 召回，不扩大候选面
  • 设置召回阈值，不强行送 LLM
  • LLM 只能二选一，不能自由生成
  • 不确定时统一 KEEP

最终结论 ​

本方案的正式定义如下：

在 ASR 生成 raw_transcript 后，对文本执行局部片段扫描；仅基于拼音表示与热词表进行语音相似度召回；对每个通过阈值的片段只保留一个 top1 热词候选；将整句以 【id|原始|候选】 形式送入 LLM；由 LLM 对每个片段执行 KEEP / REPLACE 二选一裁决；应用替换后得到 phonetic_resolved_text，再进入现有 smart clean / polished 链路。

该方案保持了实现简单、决策边界清晰、可调试、可逐步演进，适合作为 VILab 中文热词纠错能力的第一版正式方案。
