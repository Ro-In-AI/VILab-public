分支与发布规范 ​

这份文档定义的是 VILab 当前推荐的分支策略、发版节奏和公开发布仓的联动方式。

目标只有三个：

  • 日常开发和正式发布分离
  • 私有源码仓和公开分发仓分离
  • 每次正式发版都可以稳定更新官网、Release 和自动更新入口

推荐分支模型 ​

  • dev
    • 日常开发分支
    • 新功能、重构、实验性调整先进入这里
    • 可以保持较快迭代，不要求每次提交都可直接对外发布
  • main
    • 稳定分支
    • 只接收已经通过验证、准备进入正式版本的内容
    • 目标是任何时刻都尽量保持“可发布”

对于当前项目，不建议一开始就引入额外的 release/* 分支。先保持 dev + main 两层结构，已经足够清晰。

为什么不建议“合并到 main 就自动发版” ​

看起来最省事的做法是：

  • dev 合并到 main
  • GitHub Actions 自动发布安装包

但这个模式实际风险比较高，因为“进入稳定分支”和“正式对外发布”并不是一回事。

常见问题包括：

  • 版本号还没准备好
  • release notes 还没写
  • 还没完成最终回归测试
  • 这批改动已经能进 main，但想和下一两个改动一起组成一个正式版本

所以更推荐的方式是两段式：

  1. dev -> main
    • 表示代码进入稳定候选区
  2. 在 main 上打 v0.1.7 这样的 tag
    • 表示这次真的要发布

也就是说：

  • main 负责稳定
  • tag 负责发布

这是多数成熟项目更常见、也更稳的做法。

推荐发布流程 ​

日常开发阶段 ​

  • 功能开发先进入 dev
  • 在 dev 上完成联调、自测和必要的文档更新
  • 功能达到稳定状态后，再合并到 main

准备发布阶段 ​

当你准备发布一个正式版本时，建议在 main 上完成以下内容：

  1. 确认本次发布内容已经全部进入 main
  2. 更新版本号
  3. 补齐对应的 release notes 文件
  4. 确认公开站点文案和文档已同步
  5. 打正式 tag，例如 v0.1.7

正式发布阶段 ​

当 tag 推送后，GitHub Actions 应触发正式发布流程，完成：

  • 构建安装包
  • 生成签名与校验文件
  • 生成 latest.json
  • 上传到公开仓 GitHub Release
  • 更新公开站点中的下载入口和 changelog

当前项目中的职责分工 ​

私有源码仓 ​

私有仓继续承担：

  • 源代码管理
  • 日常开发与分支协作
  • 文档源文件维护
  • .github/releases/*.md 版本说明维护
  • 构建、测试、发版 workflow

公开发布仓 ​

公开仓 Ro-In-AI/VILab-public 只承担公开分发：

  • GitHub Releases
  • 安装包与签名文件
  • latest.json
  • GitHub Pages 官网
  • /download/
  • /changelog/
  • /docs/

客户端自动更新 ​

桌面应用内的 updater 固定读取公开仓：

  • https://github.com/Ro-In-AI/VILab-public/releases/latest/download/latest.json

因此，只要公开仓里的最新 Release 更新成功，客户端就能检测到新版本。

推荐的 GitHub Actions 触发方式 ​

push 到 main ​

建议只做“验证”和“站点内容同步”，不要直接发布正式版本。

适合做的事：

  • 安装依赖
  • 类型检查
  • Rust 检查和测试
  • 构建 docs
  • 构建公开站点
  • 同步公开站点内容

不建议直接做的事：

  • 创建正式 GitHub Release
  • 上传正式安装包
  • 更新 latest.json

push tag v* ​

建议把正式发布动作全部放在 tag 触发下。

适合做的事：

  • 构建 Windows/macOS 安装包
  • 上传公开仓 Release 资产
  • 生成并上传 latest.json
  • 让客户端下载到新的正式版本

这也是当前项目最推荐的正式发版入口。

一次正式发版的标准步骤 ​

以下流程适合每次发版时照着执行。

  1. 在 dev 完成功能开发和自测
  2. 将 dev 合并到 main
  3. 切到 main，确认工作区干净
  4. 更新版本号
  5. 新建 release notes，例如 .github/releases/v0.1.7.md
  6. 本地执行必要检查
  7. 提交版本相关改动到 main
  8. 打 tag，例如 v0.1.7
  9. 推送 main 和 v0.1.7
  10. 观察 GitHub Actions 是否完成
  11. 到公开仓检查 Release、latest.json、下载页和 changelog

建议的本地发布检查清单 ​

发版前至少确认下面这些项：

  • package.json 版本号正确
  • src-tauri/tauri.conf.json 版本号正确
  • .github/releases/vX.Y.Z.md 已存在
  • 文档构建通过
  • public/docs 和 public/lab 能正常生成
  • Windows 或 macOS 发布脚本本地检查通过
  • 公开仓 secrets 没过期
  • Tauri updater 签名私钥已经配置到 GitHub secrets

如果是正式外发版本，建议额外确认：

  • 安装包能正常安装
  • 客户端能正常启动
  • 自动更新说明显示正常
  • latest.json 下载地址指向公开仓
  • Release 机器能正常生成 .sig 签名文件

当前项目建议采用的规则 ​

为了降低误发版本的风险，建议固定采用下面这组规则：

  • 所有日常开发都先进入 dev
  • main 只合并准备进入正式版本的内容
  • 合并到 main 不等于立即发版
  • 只有打 tag 才算正式发版
  • 正式版本号统一使用 vX.Y.Z
  • release notes 统一写在 .github/releases/
  • 公开仓始终只承载正式稳定版本

常见问题 ​

为什么热门开源仓也常用 tag 发布 ​

因为 tag 能明确表达“这次提交对应一个正式版本”，而不仅仅是某次普通合并。

相比“只要 main 更新就发布”，tag 发布的优点是：

  • 节奏可控
  • 版本边界清楚
  • Release 页面天然和版本号一一对应
  • 回滚、追踪和排错都更容易

是否需要 release 分支 ​

当前阶段不需要。

如果未来出现这些情况，再考虑引入 release/*：

  • 同时维护多个长期版本
  • 某个大版本需要长时间冻结和修补
  • 商业版和社区版需要不同发布节奏

在目前阶段，dev + main + tag 已经是成本最低、收益最高的方案。

一句话结论 ​

VILab 当前最推荐的发布策略是：

  • dev 负责开发
  • main 负责稳定
  • tag 负责正式发布
  • 公开仓负责对外分发

这样既能保持开发效率，也能让官网、下载入口、更新日志和客户端自动更新都围绕正式版本稳定运作。
