Release 本地验收与 TDD 规范 ​

这份文档用于约束 VILab Tauri 桌面版的发布流程，目标是避免“开发环境正常，但打包后的 Release 出问题”。

基本原则 ​

  • 不要只在 tauri dev 里验证修复
  • 不要先上传 GitHub Release 再让用户帮忙验收
  • 必须先在本地构建、安装、启动并验证打包后的 Release
  • 必须保留可以解释问题和修复结果的测试、日志和截图证据
  • macOS 对外发布不能只停留在“能打出 dmg”，还必须考虑签名、公证和 Gatekeeper 体验

版本号单一来源 ​

  • package.json 是桌面端版本号的单一来源
  • 需要升版时，先改 package.json 里的 version
  • 然后运行 npm run version:sync
  • 这个脚本会把版本同步到 src-tauri/Cargo.toml、src-tauri/tauri.conf.json，并更新文档里的 /health 响应示例版本

如果只是为了发布新版本，推荐先完成版本同步，再执行后面的构建与发布命令。

推荐流程 ​

  1. 先在本地 Release 中复现问题
  2. 先补最小失败测试，再改代码
  3. 跑针对性的前端测试、Rust 测试和 smoke 测试
  4. 重新构建 Release 包
  5. 把安装包安装到一个新的临时目录，而不是直接复用旧目录
  6. 直接启动安装后的可执行文件，检查窗口、日志、数据落盘和核心用户路径
  7. 本地确认通过后，再更新 GitHub Release

平台要求 ​

  • Windows：需要保留 NSIS 或 MSI 安装包，以及 updater 相关产物
  • macOS：需要至少 ad-hoc 签名；对外发布推荐使用 Developer ID Application 证书并完成 notarization
  • macOS 对外发布前，必须实际验证下载后的 .dmg 不会被 Gatekeeper 直接拦成“已损坏”

TDD 分层 ​

不要把验证全都压在一个端到端流程上，应该分层做：

  • 前端层：补 store、client、startup 这类状态流转测试
  • Rust 层：补启动流程、资源路径、配置路由、发布态分支的回归测试
  • Release 层：保留 Windows smoke，用来验证安装、启动、重复启动和退出行为
  • macOS Release 层：验证 codesign、Gatekeeper assessment、stapler 和下载后的安装体验

Release 常见问题清单 ​

Release 与开发环境不一致时，优先检查这些问题：

  • Tauri bridge 初始化竞态
  • 主窗口隐藏后没有正确显示
  • smoke 只看进程或窗口句柄，导致假阳性
  • WebView2 用 PrintWindow 截图看起来像白屏
  • 第二个实例占用了固定端口、全局热键或单实例锁
  • 打包后资源路径与开发环境不一致
  • 开发版与 Release 共用同一个 %APPDATA% 数据目录，但 UI 同步链路有问题
  • macOS 包虽然能本地打开，但下载后的 .dmg/.app 因为没有签名或没有 notarize 被 Gatekeeper 认定为“已损坏”

验收要求 ​

本地 Release 验收至少要覆盖：

  • 主窗口真实可见，且尺寸正常，不能只看 MainWindowHandle
  • 关键页面能加载，不是空白或启动失败页
  • 历史记录、设置、模型状态、诊断信息能正确显示
  • 需要落盘的动作已经写入实际数据库或文件
  • 第二次启动不会因为端口或热键冲突而崩溃
  • macOS 下载后的安装包可以直接通过系统安全校验，而不是依赖用户手工去掉 quarantine

证据要求 ​

每次修复 Release 问题，至少保留这些证据：

  • 测试命令结果
  • smoke 测试结果
  • 本地安装目录
  • 进程 ID 与窗口尺寸
  • 打包后应用的截图
  • 关键日志
  • 涉及持久化时的数据库路径和记录数量
  • 如果是 macOS，对外发布还要保留签名、公证和 stapler 校验结果

发布门禁 ​

以下条件没有全部满足时，不要发版：

  • 失败测试已经补上并转绿
  • 本地 smoke 通过
  • 本地安装后的 Release 能正常打开
  • 目标用户路径在本地 Release 中真实可用
  • macOS 发布包已经完成签名与 notarization，或明确标注为仅限内部/本地测试
  • 有足够证据说明修复成立，而不是“理论上应该好了”
