Server-First 架构与 SenseVoice 服务化 ​

VILab 后续会从“桌面 App 内置智能能力”逐步演进为“服务端处理 + 客户端 UI”。这个方向的目标很明确：智能能力、密钥、路由和模型都放在服务端；Mac 和 Windows 客户端只负责录音、快捷键、展示结果和调用 API。

核心边界 ​

模块负责什么不负责什么
VILab ServerSTT、Provider、Routing、API Key、请求统计、文档、模型与 Worker 状态桌面窗口、托盘、快捷键、麦克风选择、剪贴板和输入法
SenseVoice Worker加载 SenseVoice 模型，接收服务端转发的音频，返回转写文本Admin API、API Key、Provider 路由、客户端 UI
VILab Client录音、交互、结果展示、复制/粘贴、配置 Server URL 和 external API keyProvider Key、本地模型、Routing、Admin Key
SDK/第三方应用调用 /v1/models 和 /v1/audio/transcriptions管理服务端、读写 SQLite、修改 Provider

一个客户端请求的路径应该是：

text
Mac/Windows 客户端
  -> VILab Server Public API
  -> Routing
  -> Groq/OpenAI 云端 Provider 或 SenseVoice Worker

客户端不应该知道 Groq Key，也不应该知道 SenseVoice 模型目录在哪里。客户端只需要知道：

text
Server URL: https://vilab.example.com
API Key: vilab_xxx
默认模型: vilab-stt-default 或 vilab-local-stt

STT 后端模式 ​

服务端现在支持两类 STT 路由：

模式Provider 示例公开模型示例适用场景
cloudgroq、openaivilab-stt-default快速部署、无需本地模型、云服务器 CPU 资源较小
server-localsensevoicevilab-local-stt模型部署在服务端，客户端只上传音频

示例：切到 Groq 云端 STT。

bash
vilab providers set groq \
  --api-key "$VILAB_GROQ_API_KEY" \
  --api-base-url https://api.groq.com/openai/v1 \
  --stt-model whisper-large-v3-turbo \
  --enable

vilab routing set \
  --stt-provider groq \
  --public-model vilab-stt-default \
  --transcription-model whisper-large-v3-turbo

示例：切到服务端本地 SenseVoice。

bash
vilab providers set sensevoice \
  --api-base-url http://vilab-sensevoice:9877 \
  --stt-model sensevoice-small \
  --enable

vilab routing set \
  --stt-provider sensevoice \
  --public-model vilab-local-stt \
  --transcription-model sensevoice-small

当 provider 是 sensevoice 时，vilab routing set 会默认使用 server-local。如果你要显式写，也可以这样：

bash
vilab routing set \
  --stt-mode server-local \
  --stt-provider sensevoice \
  --public-model vilab-local-stt \
  --transcription-model sensevoice-small

SenseVoice Worker 为什么独立 ​

SenseVoice 不直接塞进 vilab-server 主进程，而是单独运行 vilab-sensevoice-worker。这样做有几个好处：

  • 主服务仍然轻量，启动快，Admin API 和 Public API 不被模型加载拖慢。
  • CPU 和 GPU 镜像可以分开演进，后续可以增加 docker compose --profile gpu up -d。
  • Worker 崩溃或重启时，不影响 Admin API、API Key 管理和文档服务。
  • 将来可以替换成 Python/FastAPI、sherpa-onnx、FunASR 或其它更成熟的推理适配器，只要内部协议保持一致。

当前 Worker v1 提供内部协议：

http
GET /health
GET /models
POST /models/{id}/download
POST /models/{id}/activate
POST /reload
POST /transcribe

POST /transcribe 返回：

json
{
  "text": "转写文本",
  "model": "sensevoice-small",
  "durationMs": 1234
}

当前 Rust Worker 分成两种构建方式：普通 server feature 只保留协议、健康检查、模型目录检查和 mock 冒烟转写；Docker 镜像里的 vilab-sensevoice-worker 会启用 SenseVoice runtime，模型目录存在时可以直接调用 transcribe-rs 的 ONNX SenseVoice 推理。后续仍然可以把它替换成 Python/FastAPI、sherpa-onnx、FunASR 或 GPU 专用 worker，只要内部协议保持一致。

Docker 拓扑 ​

docker-compose.server.yml 会启动两个服务：

text
vilab-server
  - 对宿主机暴露 9876
  - 保存 SQLite、Admin Key、external API key
  - 通过 VILAB_SENSEVOICE_URL 访问 worker

vilab-sensevoice
  - 容器内监听 9877
  - 挂载 vilab-models:/models
  - 不直接暴露给公网

启动：

bash
docker compose -f docker-compose.server.yml up --build -d

查看服务端状态：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab doctor

查看 SenseVoice 模型状态：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab models list

如果只是验证链路，可以开启 mock 转写：

bash
VILAB_SENSEVOICE_MOCK_TEXT="这是一段 SenseVoice 冒烟转写" \
docker compose -f docker-compose.server.yml up --build -d

然后配置 SenseVoice 路由：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab init

在交互向导里选择 Local: SenseVoice。向导会显示 worker 是否可达、模型是否安装，并进入 选择本地模型 步骤；默认会优先选中当前 active 模型，其次选中已安装模型，最后回退到 sensevoice-small。完成后，客户端仍然只调用：

bash
curl -H "Authorization: Bearer $VILAB_API_KEY" \
  -F file=@sample.wav \
  -F model=vilab-local-stt \
  http://127.0.0.1:9876/v1/audio/transcriptions

模型目录 ​

SenseVoice Worker 默认读取：

text
/models/sensevoice-small

在 Docker Compose 里，这个目录来自 volume：

yaml
volumes:
  - vilab-models:/models

如果模型没有安装，vilab init 选择 Local: SenseVoice 后会提示是否安装；确认后会显示下载进度条，把 model.int8.onnx 和 tokens.txt 下载到 /models/sensevoice-small 并激活模型。默认模型会使用内置文件大小显示真实百分比；自定义镜像源拿不到总大小时，也会显示“总大小未知”的活动进度条和已下载体积。也可以手动运行：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab models download sensevoice-small
docker compose -f docker-compose.server.yml exec vilab-server vilab models activate sensevoice-small

模型目录存在且必需文件齐全后，Docker worker 会进入 runtime 状态并尝试真实推理。如果服务器无法访问默认下载源，可以设置 VILAB_SENSEVOICE_MODEL_BASE_URL 指向内部镜像。

Client Remote-First ​

桌面客户端后续主路径应该变成：

  1. 用户填写 Server URL。
  2. 用户填写自己的 external API key。
  3. 客户端测试 /health。
  4. 客户端用 external API key 测试 /v1/models。
  5. 录音后上传到 /v1/audio/transcriptions。
  6. 展示转写结果，并执行复制、粘贴或快捷键动作。

客户端不再保存 Provider API Key，也不再直接管理 SenseVoice 模型。过渡期可以保留旧的单机离线能力，但应放到“单机离线模式”或“Legacy”区域，不作为默认入口。

v1 边界 ​

当前实现已经把 server-local 的协议、路由、CLI、Docker sidecar、runtime worker 构建和文档闭环搭起来。仍然需要后续继续完成的部分是：

  • 模型下载、校验、激活的完整受控流程。
  • GPU compose profile。
  • 客户端设置页的 Remote-first 改造。
  • 将 rewrite、prompt、dictionary/hotword 等文本处理能力继续迁移到服务端。
