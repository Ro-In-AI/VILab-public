无头服务器 ​

VILab 无头服务器是面向 Ubuntu x64 云服务器的单租户自托管服务。它保留 OpenAI 兼容的公开 API，通过 Admin API 或 vilab 管理，并开始向 Server-First 架构演进：云端 STT 走 Provider，server-local STT 走服务端本地的独立 SenseVoice worker sidecar。

它不会启动 Tauri，不创建窗口，不注册快捷键，不访问剪贴板、麦克风或输入法，也不挂载 Prompt Lab 和输入法管理等桌面专属接口。

Docker 快速开始 ​

bash
export VILAB_GROQ_API_KEY="gsk_..."
docker compose -f docker-compose.server.yml up --build

docker-compose.server.yml 默认启动 vilab-server 和 vilab-sensevoice 两个服务。vilab-server 容器内部监听 0.0.0.0:9876，宿主机默认映射到 127.0.0.1:9876；SQLite、Admin Key、external API key 和 provider secrets 会保存在 /data volume。vilab-sensevoice 容器内部监听 9877，模型目录挂载到 /models，默认不直接暴露给宿主机或公网。

bash
curl http://127.0.0.1:9876/health

本地代码更新后，推荐用部署脚本重建并替换容器：

bash
npm run server:docker:deploy

这个脚本会执行 Docker Compose 构建和重建容器，但不会删除 volume，所以 /data 里的 SQLite、Admin Key、external API key 和 Provider 配置会保留。脚本还会等待 /health 通过，并检查容器内的 vilab CLI 是否已经包含最新能力。

如果你像当前开发环境一样把服务端映射到其它端口，可以显式指定：

bash
npm run server:docker:deploy -- --host-port 19876

浏览器打开文档入口：

text
http://127.0.0.1:9876/docs/

无头服务端复用桌面客户端同一份 VitePress 文档构建产物和同一套静态文件服务逻辑。Docker 镜像默认把文档放在 /usr/local/share/vilab/docs，并由 vilab-server 挂载到 /docs/。

如果宿主机 9876 已经被桌面 App 或其它进程占用，可以换一个宿主机端口。容器内部仍然使用 9876，只有对外访问地址变化：

bash
VILAB_HOST_PORT=19876 docker compose -f docker-compose.server.yml up -d

然后访问：

text
http://127.0.0.1:19876/docs/

如果服务端部署在云服务器或反向代理后面，建议显式设置公开地址：

bash
VILAB_PUBLIC_URL="https://api.example.com" docker compose -f docker-compose.server.yml up -d

VILAB_PUBLIC_URL 会用于 vilab status、vilab init 初始化摘要和客户端配置提示。

首次启动时，如果没有提供 VILAB_ADMIN_KEY，服务端会自动生成 Admin Key，并把完整值写入数据目录：

bash
docker compose -f docker-compose.server.yml exec vilab-server cat /data/initial-admin-key.json

首次启动时，如果数据库里还没有 external API key，服务端会自动创建一把 default key，并把完整值写入数据目录：

bash
docker compose -f docker-compose.server.yml exec vilab-server cat /data/initial-api-key.json

拿到 Admin Key 之后，也可以通过 CLI 查询初始 external API key：

bash
vilab keys initial

这个初始 key 可以直接发给第一台客户端做连接测试。普通 vilab keys list 只显示 preview；如果完整 key 丢失，建议创建新 key，而不是依赖长期明文查询。

配置 ​

vilab-server 读取这些环境变量：

变量默认值作用
VILAB_DATA_DIR$HOME/.vilab-serverSQLite 与 provider-secrets.env 目录
VILAB_BIND127.0.0.1:9876HTTP 监听地址
VILAB_DOCS_DIR自动探测/docs/ 静态文档目录
VILAB_HOST_PORT9876Docker 宿主机映射端口
VILAB_PUBLIC_URL自动推断客户端应该访问的公开服务地址
VILAB_ADMIN_KEY本地开发自动生成/admin/* 的 Bearer token
VILAB_GROQ_API_KEY未设置Provider API Key 环境变量覆盖
VILAB_SENSEVOICE_URLhttp://127.0.0.1:9877SenseVoice worker 地址；Docker Compose 默认指向 http://vilab-sensevoice:9877
VILAB_SENSEVOICE_MODELsensevoice-smallWorker 默认模型 ID
VILAB_SENSEVOICE_MODEL_BASE_URLHuggingFace 官方模型地址SenseVoice 模型下载源；网络受限时可以改成内部镜像
VILAB_SENSEVOICE_MOCK_TEXT未设置协议冒烟测试用 mock 转写文本

也可以通过 vilab providers set ... --api-key ... 写入 Provider API Key；密钥会落在数据卷内的 provider-secrets.env。

CLI 管理 ​

构建或安装 vilab 后，指向服务器：

bash
export VILAB_SERVER_URL="http://127.0.0.1:9876"
export VILAB_ADMIN_KEY="<initial-admin-key.json 里的 adminKey>"

vilab init
vilab status
vilab providers list
vilab providers set groq --api-key "$VILAB_GROQ_API_KEY" --stt-model whisper-large-v3-turbo --enable
vilab routing set --stt-provider groq --public-model vilab-stt-default --transcription-model whisper-large-v3-turbo
vilab keys initial
vilab keys create --name "demo app"
vilab models list
vilab doctor

vilab init 是推荐的首次配置入口。在交互式终端里，它会显示选择式向导：自动读取 initial-admin-key.json、用方向键选择 STT Provider、用隐藏输入填写云端 Provider API Key、确认是否创建新的客户端 API Key，并自动跑 /health、/admin/status 和 /v1/models 冒烟检查。

如果选择 Local: SenseVoice，向导会切换成本地模型流程：先检查 /admin/workers 和 /admin/models，显示 SenseVoice worker 是否可达、模型是否安装，然后让你从服务端返回的本地模型列表中选择，例如 sensevoice-small。如果模型尚未安装，向导会询问是否现在安装；确认后会创建下载任务，使用进度条显示已下载体积、总大小和当前文件，下载 model.int8.onnx 与 tokens.txt 后再自动激活模型。默认 SenseVoice 模型内置了官方文件大小，正常会直接显示真实百分比进度；如果自定义下载源暂时没有返回总大小，CLI 仍会显示“总大小未知”的活动进度条和已下载体积，拿到 content-length 后再切换为真实下载进度条。这时不需要填写 Provider API Key，也不会再询问云端改写模型；路由会自动使用 server-local 和 vilab-local-stt。

如果你已经进入 Docker 容器，通常不需要手动 export Admin Key：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab init

如果命令运行在非交互环境，例如 CI、脚本或没有 TTY 的 SSH 命令里，请使用 --yes 并补齐必要参数。

脚本或 CI 可以使用非交互形式：

bash
vilab init --yes \
  --provider groq \
  --provider-api-key "$VILAB_GROQ_API_KEY" \
  --public-model vilab-stt-default \
  --skip-transcribe

如果希望 init 同时创建一把新的客户端 key，可以显式传入：

bash
vilab init --yes \
  --provider groq \
  --external-key-name "alice-macbook" \
  --skip-transcribe

任意命令加 --json 可以输出机器可读 JSON。

vilab doctor 用来诊断连接、路由、端口和 worker。它会检查 /health、/admin/status、/admin/routing、/admin/workers，并显示当前服务端认为的公开访问地址、文档地址、模型列表地址，以及端口冲突时推荐的 VILAB_HOST_PORT 命令。

SenseVoice 服务端本地模式 ​

如果想让服务端使用 SenseVoice，而不是云端 Provider，可以让路由指向 sensevoice：

推荐直接使用初始化向导：

bash
docker compose -f docker-compose.server.yml exec vilab-server vilab init

在 选择 STT Provider 时选择 Local: SenseVoice，再选择本地模型；如果模型没有安装，按提示确认安装即可。非交互脚本默认会尝试安装缺失模型，可以写成：

bash
vilab init --yes \
  --provider sensevoice \
  --stt-model sensevoice-small \
  --public-model vilab-local-stt \
  --skip-transcribe

如果服务器无法访问 HuggingFace，可以先配置模型下载镜像：

bash
VILAB_SENSEVOICE_MODEL_BASE_URL="https://mirror.example.com/sensevoice-small" \
npm run server:docker:deploy

如果你只想保存路由，不希望初始化时下载大模型，可以加：

bash
vilab init --yes --provider sensevoice --skip-model-install --skip-transcribe

也可以拆成两步手动配置：

bash
vilab providers set sensevoice \
  --api-base-url http://vilab-sensevoice:9877 \
  --stt-model sensevoice-small \
  --enable

vilab routing set \
  --stt-provider sensevoice \
  --public-model vilab-local-stt \
  --transcription-model sensevoice-small

当 provider 是 sensevoice 时，vilab routing set 会默认把 sttMode 设置为 server-local。客户端调用方式不变，只是模型名改成服务端公开出来的别名：

bash
curl -H "Authorization: Bearer $VILAB_API_KEY" \
  -F file=@sample.wav \
  -F model=vilab-local-stt \
  http://127.0.0.1:9876/v1/audio/transcriptions

查看 worker 和模型状态：

bash
vilab doctor
vilab models list

如果你只是想验证 server -> worker -> public API 的链路，可以用 mock 文本启动：

bash
VILAB_SENSEVOICE_MOCK_TEXT="SenseVoice mock 转写成功" \
docker compose -f docker-compose.server.yml up --build -d

当前 worker v1 支持健康检查、模型目录检查、模型下载、mock 冒烟转写和服务端转发。Docker 镜像里的 worker 会启用 SenseVoice runtime；当 /models/sensevoice-small 下存在 model.int8.onnx 和 tokens.txt 时，会通过 transcribe-rs 调用 ONNX SenseVoice 推理。GPU profile 会继续作为后续增强。

公开 API ​

公开 API 需要 Authorization: Bearer <external-api-key>。普通客户端和 SDK 默认连接远程 VILab Server，只使用 external API key；Admin Key 只用于 /admin/* 管理接口，不要发给客户端。

bash
curl -H "Authorization: Bearer $VILAB_API_KEY" \
  http://127.0.0.1:9876/v1/models

curl -H "Authorization: Bearer $VILAB_API_KEY" \
  -F file=@sample.wav \
  -F model=vilab-stt-default \
  http://127.0.0.1:9876/v1/audio/transcriptions

SDK 示例 ​

TypeScript:

ts
import { VILabClient } from '@vilab/client';
import { readFile } from 'node:fs/promises';

const client = new VILabClient({
  serverUrl: process.env.VILAB_SERVER_URL ?? 'https://vilab.example.com',
  apiKey: process.env.VILAB_API_KEY!,
});
const result = await client.transcribe(await readFile('sample.wav'));

Python:

python
from vilab_client import VILabClient
import os

client = VILabClient(
    os.environ.get("VILAB_SERVER_URL", "https://vilab.example.com"),
    os.environ["VILAB_API_KEY"],
)
print(client.transcribe("sample.wav"))

v1 边界 ​

服务端仍然拒绝桌面旧语义里的 sttMode=local。服务端本地模型使用新的 sttMode=server-local，由 vilab-server 调用独立 worker 完成。

Linux 桌面打包、多租户组织/项目、网页管理后台，都属于后续独立方向。GPU 版 SenseVoice worker 会以后续 compose profile 形式加入。

无头服务端会挂载 /docs/ 文档入口，但不会挂载 /lab/ Prompt Lab 或 /admin/ime/* 桌面输入法管理接口。
